Validating Forms

  • We use client-side form validation to verify that all required form controls are filled out and in the correct format.
  • It is often an important feature of good user experience, by catching invalid data on the client-side. If it gets to the server and it is rejected there is a noticable delay due to the client and server communication.
  • Client-side validation should not be considered an exhaustive security measure. Both the server, and the client should perform security checks.

Why do we validate forms?

  • We want to get the right data, in the right format.
  • We want to protect our user's data.
  • We want to protect ourselves.

Different types of client-side validation

  1. Built-in form validation
  2. JavaScript

Using built-in form validation

  • We can validate most user data without relying on JavaScript. There are a list of validation attributes:
  • required (specifies whether a form field needs to be filled in before a form can be submitted)
  • minlength and maxlength (specifies the minimum and maximum length of textual data)
  • min and max (specifies the minimum and maximum values of numerical data)
  • type (specifies the data type required)
  • pattern (regular expressions)
  • Depending on rules we set, the input from our form can either set a CSS psuedo-class to :valid or :invalid. Here we can use CSS to change the display of our page and inform users of what is valid or invalid input.

Using Fetch

  • The Promise returned from fetch will not reject on HTTP error status
  • It is important that we check response status to diagnose errors
  • Fetch() will not send cross-origin cookies unless we set the credentials init option
